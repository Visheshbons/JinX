# JinX Assembly (JASM) Documentation

JASM is the current text output format emitted by the JinX compiler backend.

As of **April 26, 2026**, JASM generation is implemented for a small, bootstrapping subset of JC (includes, function declarations, and simple local variable declarations).

---

## 1) Current Compilation Flow

Current flow in this repository:

```text
JC source (.jc) -> lexer.py -> parser.py AST -> generator.py -> JASM (.jasm)
```

`OS/compiler/main.py` now writes a `.jasm` file next to the input `.jc` file after parsing and JASM generation succeed.

---

## 2) File Extension and Output Location

- Extension: `.jasm`
- Output path: generated by replacing `.jc` with `.jasm` in the input file path.

Example:
- Input: `OS/kernel/main.jc`
- Output: `OS/kernel/main.jasm`

---

## 3) Syntax Style Emitted Today

The current generator emits Intel-like assembly text with:

- `global <symbol>` declarations (without a leading dot)
- Function labels (`<symbol>:`)
- Standard function prologue/epilogue using `rbp`/`rsp`
- Size-qualified memory stores (`byte`, `dword`, `qword`)
- Semicolon comments (`; ...`)

Example (current real output shape):

```asm
global k_main
k_main:
  push rbp
  mov rbp, rsp
  ; u32 screen = 0xB8000;
  mov dword [rbp-4], 0xB8000
  mov rsp, rbp
  pop rbp
  ret
```

---

## 4) What JC Constructs Are Lowered Right Now

### 4.1 `#include`

Parser creates `INCLUDE` nodes; generator emits them as comments only:

```asm
; included #include <...>
```

No symbol importing or directive emission is performed yet.

### 4.2 Function declarations

For each `func <name>(...) <ret> { ... }`, generator emits:

1. `global <name>`
2. `<name>:` label
3. Prologue:
   - `push rbp`
   - `mov rbp, rsp`
4. Generated body statements (currently variable declarations only)
5. Epilogue:
   - `mov rsp, rbp`
   - `pop rbp`
   - `ret`

### 4.3 Local variable declarations

Supported declaration types in parser body handling:

- `u32`
- `u8`
- `ptr`
- `str` (accepted by parser)

The generator currently maps storage width as:

- `u32` -> 4 bytes (`dword`)
- `u8` or `bool` -> 1 byte (`byte`)
- `ptr` -> 8 bytes (`qword`)
- Anything else (including current `str`) -> fallback 8 bytes (`qword`)

Each local is stack-assigned at a negative `[rbp-<n>]` offset in declaration order as the stack frame grows downward.

---

## 5) Current Limitations

This is an intentionally minimal backend. Important current limitations:

- No `.section`, `.extern`, `.data`, or string directives are emitted yet.
- No expression codegen (declarations only store a single literal/identifier token).
- Function arguments are parsed but not yet used in codegen.
- Return statements are tokenized in lexer but not lowered in parser/generator body logic.
- No register allocation, control flow lowering, or calling convention enforcement.
- Output is textual JASM only; no machine code emission.
